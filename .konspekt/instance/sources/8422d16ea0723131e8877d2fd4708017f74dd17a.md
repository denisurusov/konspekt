# Source excerpt — provenance decision (2026-06-22)

Conversation: konspekt / "provenance as chat text". Captured push-based at
extraction time. This is the addressable source text the provenance-model
entities were extracted from; this file's git blob SHA is the `contentHash`
they carry.

## Human — opening
given we can't get to the internally kept IDs of conversations, attach actual
chat text (my prompts and your responses) as provenance mechanism? I think all
types of nodes would benefit from this

## Human — options + git
the choices are 1. assume the vendors will expose deep links to the relevant
parts of conversations or 2. they will serve up hashes that can be later use to
find the parts or 3. bring the parts in "ourselves". We *could* cheat a little
and use git and its commit hash mechanism to store the parts; (you'll be a good
vendor and extract the parts for now). Specification-wise, we'll just say "a
node has to have a pointer to an addressable text and the hashes on that text
have to match the hash stored at the construction time"

## Human — ordering + fields
ordering will be solved by timestamps. the "parts" is not very deterministic,
but then again - nothing in LLM space is and, the worst case, we'll add UI
manual curation support. We do want source ref and content hash, as long as
source ref points me to the files you put in git

## Human — keep timestamp
we need to keep timestamp. github knows it, but I don't want to rely on github
for timekeeping

## Synthesis (assistant, condensed)
- Provenance anchors on source TEXT via sourceRef + contentHash, not host
  conversation/message ids — the maintainer is blind to those.
- Two fields, address vs digest, kept distinct; they coincide in the git
  binding where the blob SHA is the content hash.
- Push-based ingestion: write source excerpts into the store at extraction
  time; idempotence becomes match-level dedup on the content-addressed source;
  the watermark demotes to a timestamp-ordered optimization; the edited-message
  special case retires (different text -> different hash -> different source).
- timestamp is source time, binding-independent; never read git commit time.
