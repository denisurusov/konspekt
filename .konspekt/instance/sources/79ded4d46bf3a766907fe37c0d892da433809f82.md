## Human — github now, MCP next?
it currently works with github-based approach, which is fine for a prototype. As
the next step - would it make sense to create an MCP server and add to something
like connectors in Claude?

## Human — central service, two ways in, verbs unchanged
I am thinking that if/when I decide to stand up a centralized service
w/datastore, I can offer two ways of connecting to it - APIs and MCP. MCP seems
better because of native integration with all the LLM vendors. The verbs stay as
they are, only instead of "sync to github" the meaning would be "sync to
konspekt mcp"

## Assistant — MCP rides over the API, it is not an alternative to it
"MCP isn't an *alternative* to an API — it sits *over* one." If you stand up a
central service with a datastore, "you'll have a core service layer regardless;
REST and MCP are two *adapters* over that same core, not two competing back
ends." The right mental model "isn't "API or MCP," it's "core service, MCP as
the LLM-facing skin, REST as the everything-else skin.""

## Assistant — the neutral public read is load-bearing for conformance
"the REST/raw-markdown path isn't the lesser sibling you offer as a courtesy.
It's load-bearing for conformance." "An MCP-only central service is exactly that
danger: state reachable only through LLM tool calls is *public-ish* but it fails
the probe — a human can't export-and-paste the instance without going through
your tool surface. The thing that disarms the lock-in is a plain neutral read —
raw markdown over a URL, no model in the loop." So "offering 'both' isn't
redundancy. The non-MCP path is what proves the MCP binding isn't capture."

## Assistant — keep the central store dumb
"keep the central store **dumb** — `read`/`write` only." "The MCP server should
expose the verbs as orchestration over read/write, not start adjudicating
proposals itself — the moment it does, you've broken 'review lives in the
conversation' and rebuilt the thing you're designing against."

## Assistant — vendor reach is real but uneven at the consumer-connector layer
The developer-facing reach of MCP is "near-universal; the "add it in the
settings panel like a connector" reach is real on some surfaces and
developer-only on others." MCP itself got there by doing what konspekt is trying
to do — "legible, vendor-neutral, handed to a neutral foundation rather than
defended."
