verbs + model (while things like provenance is an implementation constraint) and

## Assistant — verbs and model live together; provenance is the snag
"The verbs and the model do live together — both are in
`spec/data-model/SPEC.md` (the verbs are the 'Human vocabulary (v1)' section
right alongside the entity/edge model). So 'two things' is the right conceptual
lens, and structurally they're the one portable vocabulary."

"But provenance is the snag. The data-model spec, Principle 3 ('Auditable by
construction'), puts provenance inside the model as a requirement ... So
provenance isn't wholly an implementation constraint — taken literally,
'provenance is an implementation constraint' contradicts your own SPEC."

## Assistant — the clean split (model requires, binding decides how)
"Model requires auditability — every entity carries provenance and a
proposed → accepted state. That's normative."

"Binding decides how — content-addressing via git blob SHAs, the sources/ store,
push-based ingestion, the verify probe. That's the implementation constraint,
and it lives in spec/architecture/."

## Assistant — the two-pillars framing
"konspekt is two things, both small and both in spec/data-model/: A model — a
typed knowledge graph of a project ... A verb set — the small authority
vocabulary a human issues over that graph: pin, validate, refute, resolve,
abandon, lift. ... Everything else is an implementation constraint — not new
vocabulary, but what a conformant binding must do to keep that graph true and
carry it between platforms."
