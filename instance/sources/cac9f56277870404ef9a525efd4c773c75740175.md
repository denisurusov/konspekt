# Source excerpt — provenance decision (2026-06-22)

Conversation: konspekt / "provenance as chat text". Captured push-based at
extraction time: verbatim spans from BOTH sides of the exchange the
provenance-model entities were extracted from. This file's git blob SHA is the
`contentHash` those entities carry. (Supersedes an earlier excerpt,

## Human — opening
given we can't get to the internally kept IDs of conversations, attach actual
chat text (my prompts and your responses) as provenance mechanism? I think all
types of nodes would benefit from this

## Human — options + git
the choices are 1. assume the vendors will expose deep links to the relevant
parts of conversations or 2. they will serve up hashes that can be later use to
find the parts or 3. bring the parts in "ourselves". We *could* cheat a little
and use git and its commit hash mechanism to store the parts; (you'll be a good
vendor and extract the parts for now). Specification-wise, we'll just say "a
node has to have a pointer to an addressable text and the hashes on that text
have to match the hash stored at the construction time"

## Assistant — git is content-addressed; address and hash are two fields
"git *is* a content-addressed object store, so if we bring the parts in as
blobs, the blob SHA already *is* the content hash. We don't define a hashing
function, we don't maintain a `Source` entity with its own digest machinery, and
'the hash matches' stops being a check we run — it's structural."

"Address and hash want to be two fields, even though git tempts you to collapse
them into one." Deep link: the text "can drift ... So you *must* store an
independent construction-time hash to detect drift. Address ≠ hash." Git: "the
SHA *is* the content hash. Collapses cleanly."

Verify probe: "an entity's provenance verifies iff hashing the text resolved at
its `sourceRef` reproduces its stored `contentHash`."

## Human — ordering + fields
ordering will be solved by timestamps. the "parts" is not very deterministic,
but then again - nothing in LLM space is and, the worst case, we'll add UI
manual curation support. We do want source ref and content hash, as long as
source ref points me to the files you put in git

## Assistant — push-based inverts idempotence; the source is the stable layer
"Bringing parts in is *push-based* ingestion: at extraction time I write the
source excerpt into git alongside the atoms. That means I never re-read a
durable already-consumed source, so the skip-watermark — the thing that needed
an orderable key — stops being load-bearing for correctness. What carries
correctness instead is match-level dedup on the source blob..."

"the *source excerpt itself* is near-deterministic, because copying text is
close to deterministic even when interpretation isn't. So anchoring identity on
the source blob's hash gives you a stable dedup key precisely where atom
extraction is shaky. The non-determinism you're conceding sits one layer above
the layer that needs to be stable."

## Human — keep timestamp
we need to keep timestamp. github knows it, but I don't want to rely on github
for timekeeping

## Assistant — timestamp is source time, not transport time
"commit time is the *transport* recording when a write landed, and
`provenance.timestamp` is the *source* fact of when the exchange happened. Those
genuinely differ ... Reading commit time would make the timeline mean 'when we
persisted' instead of 'when it happened'."

"`timestamp` stays a first-class, binding-independent provenance field: source
time, carried by the data, identical on every binding."
